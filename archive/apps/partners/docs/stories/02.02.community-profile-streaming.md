# Story 02.02: Community & Messaging Streaming

## Status
Backlog

## Story
**As a** community-first partner,
**I want** all Community/Messaging routes to paint instantly,
**so that** I can browse channels, profiles, and help docs without waiting 10–16 seconds for mock data to hydrate.

## Acceptance Criteria

1. **Performance Budget**
   - `/partners/community/profile/[profileId]`, `/community/announcements`, `/community/wins`, `/community/help`, `/community/messages` each ≤ 3,000 ms load / ≤ 2,000 ms FCP (baseline: 8.7–15.9 s in `route-metrics.tsv`).
   - `/partners/community` index, `/community/all-channels`, `/community/all-partners` ≤ 2,500 ms load.

2. **Server-Side Data Streaming**
   - Partner directory + help center data fetched via server components (`generateStaticParams` + `cache(fetch)`), no inline megabyte arrays inside client files.
   - Profile page hero + stats render on server, with only interactive widgets (`MessageButton`, `InviteButton`) as client components.

3. **Lazy Hydration for Messaging UI**
   - Chat composer, directory overlays, and DM thread panels hydrate only on interaction/visibility (use `IntersectionObserver` or focus listeners).
   - Channel skeletons appear immediately while `dynamic()` imports load.

4. **Measurement & Regression Guard**
   - New baseline run proves all targeted routes meet budgets; results noted in story log + checklist.
   - Add per-route `loading.tsx` + `error.tsx` to prevent blank screens if data fetch fails.
