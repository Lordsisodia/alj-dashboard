# Story 02.04: Pipeline & Recruitment Hydration Control

<!-- Powered by BMAD™ Core -->

## Status
Backlog

## Story
**As a** partner managing deals and recruits,
**I want** pipeline/recruitment pages to stay under 1.2 s even with wizard mock data,
**so that** the UI feels instant while we build backend logic later.

## Acceptance Criteria

1. **Performance Budget**
   - `/partners/pipeline-ops`, `/pipeline-ops/active-deals`, `/pipeline-ops/client-notes`, `/pipeline-ops/prospects`, `/pipeline-ops/submit-client` load ≤ 1,200 ms / FCP ≤ 900 ms (baseline 0.4–6.5 s in TSV rows 48–55).
   - `/partners/recruitment`, `/recruitment/prospects`, `/recruitment/team`, `/recruitment/dashboard` load ≤ 1,200 ms.

2. **Externalized Wizard Configs**
   - All large config arrays (`wizardPrompts`, recruitment notes) moved to JSON under `public/data/pipeline` or `public/data/recruitment` and streamed via server helpers.
   - `getPipelineOpsOverview` & recruitment data loaders return lean summaries on initial render (≤ 25 items).

3. **Lazy Hydration & Interaction Gating**
   - Submit Client wizard, recruitment profile overlays, and composers hydrate on interaction only (use `useHydrateOnView` + `IntersectionObserver`).
   - Directory overlays + heavy charts moved to `next/dynamic` with suspense fallbacks.

4. **Measurement & Regression Guard**
   - Perf sweep proves all listed routes within budget; results logged + checklist updated.
   - Add route-level skeletons (`loading.tsx`) for pipeline/recruitment indexes.
